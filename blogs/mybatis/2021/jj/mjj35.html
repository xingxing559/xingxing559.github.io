    # 35. 二次封装-需求与思路设计

    小册的最后一个部分，我们来一起给 MyBatis 做一个二次封装，这也是阿熊第一次尝试着带各位自己动作做一些封装工作。对框架的二次封装，要尽可能的考虑易用性、通用性、可扩展性等。

    下面我们先聊一下此次封装的需求。

    # 1. 需求说明

    我们要做的是能尽可能用简单的办法，封装出一套可以自动单表增删改查的底层。除此之外，可以设计并提供一些实用特性，例如乐观锁、逻辑删除等。

    # 1.1 单表增删改查

    既然要做二次封装和自动化，最起码的单表增删改查是必不可少的。我们对于单表增删改查的封装预期，可以列举为如下几个特性：

    • 不再需要编写相同 / 相似的方法、statement （如 insert update delete findAll findById ）
    • 更自由的单表查询策略，可以做到尽可能的不写 statement

    # 1.2 特性封装

    我们现在接触的比较多的 MyBatis 增强框架都包含一些比较实用的特性，我们也可以来实现一下。本次我们要做的特性包括：

    • 主键生成器
    • 乐观锁
    • 逻辑删除
    • 分页插件（引入 PageHelper ）

    当然，如果小伙伴有什么好的 idea ，也可以尽情发挥 ~

    # 2. 拆解思考

    我们想一下，如果要实现通用的单表 CRUD ，是不是需要收集 / 定义一些关键的信息：

    • 表的属性
    • 列的属性
    • 实体模型类中属性与列的对应
    • 主键策略
    • 属性更新策略
    • 等等

    这些信息我们如何获得 / 定义呢？

    抽取通用逻辑的话，Mapper 应该得有个父接口，这个父接口又该怎么定义呢？接口对应的 statement 的实现写在哪里呢？怎么写呢？

    如果要扩展一些高级特性，这些特性如何跟通用的逻辑相配合呢？

    OK ，带着这些问题，我们准备出发开工。

    # 3. 环境搭建

    下面我们先把环境搭建一下。

    # 3.1 导入依赖

    先来创建一个新的项目模块吧，前面已经 8 个模块了，所以咱这个模块名叫 mybatis-09-encapsulate 吧。
