    兴兴兴兴吖 2021/12/20 玩转 MyBatis：深度解析与定制

    # 12. 注解-Provider系列注解

    上一章我们接触了 MyBatis 的基本 CRUD 注解，它们都是通过编写简单的 SQL 来代替 mapper.xml 中的 statement 。除此之外，MyBatis 还提供了 4 种 Provider 注解，使用它们同样可以实现 CRUD 操作。本章我们就来学习一下它。

    在开始学习之前小册先提前声明一句：Provider 注解在实际开发中或许不会用到（甚至注解 statement 定义都不会搞），但 Provider 型注解的一个好处是更灵活，巧妙的利用它可以起到不错的效果，后面在二次封装的部分我们还会讲到它的。

    # 1. @SelectProvider

    所有的 Provider 型注解，都有两个属性：type 和 method ，它们分别指代一个提供 SQL 的类和方法，被引用的提供 SQL 语句的方法都需要有一个 String 类型的返回值，参数可以根据实际情况定义。下面我们可以先举几个简单的例子。

    # 1.1 基本使用

    本章的内容我们都写在 UserAnnotationMapper 中吧！我们还是先来定义一个 findAll 方法，并标注 @SelectProvider 注解：

    public interface UserAnnotationMapper {
    
    @SelectProvider(type = UserMapperProvider.class, method = "findAll")
    List<User> findAll();
}

    # 1.2 复杂SQL的拼装

    本身 @SelectProvider 注解的使用是简单的，不过里面如何拼装 SQL ，这个我们可以再多接触一点。

    比方说，我们要实现如下的 SQL ：

        <select id="findAll" resultType="User">
        select * from tbl_user
        <where>
            <if test="id != null">
                and id = #{id}
            </if>
            <if test="name != null and name != ''">
                and name like concat('%', #{name}, '%')
            </if>
        </where>
        order by id asc
    </select>

        public String findAllByExample(User example) {
        SQL sql = new SQL();

        if (example.getId() != null) {
            sql.AND().WHERE("id = #{id}");
        }
        if (example.getName() != null && example.getName().trim().length() > 0) {
            sql.AND().WHERE("name like concat('%', #{name}, '%')");
        }
        sql.ORDER_BY("id asc");
        return sql.toString();
    }

    这里小册多提一句，可能有部分小伙伴知道有一个 API 叫 SelectBuilder ，这个家伙也能实现动态 SQL 的拼接，不过翻看新版的 MyBatis 源码，可以发现这个类已经过时了，javadoc 中也提示换用 SQL 这个类来构造。SelectBuilder 它本身是定义了一组静态方法，配合一个 ThreadLocal 实现 SQL 拼接，不过这样设计不是特别好，MyBatis 后来新造了这个 SQL 类，就把之前的 SelectBuilder 标记为过时了，毕竟基于方法内部 new 出来的对象的操作，线程安全问题会少很多。

    # 1.3 使用时要注意的

    尽管看上去使用动态 SQL 的这个 API 很酷炫（并没有？），不过话又说回来，要使用 Provider 系列注解的时候，这个提供 SQL 语句的 Provider 类是有几个注意事项的：

    • 必须有无参构造器（不存在则会报构造器找不到的异常）
    • Mapper 接口中的参数列表跟 Provider 类中的方法参数列表原则上一致

    第一个注意的不用小册多解释，小伙伴们可以自行测试一下；第二个注意事项小册可以解释一下：默认情况下，我们在调用 Mapper 接口传入参数时，MyBatis 发现我们使用了 Provider 系列的注解，会把这些参数直接转交给 Provider 类的方法，让它来负责生成 SQL 语句，如果这个时候方法参数列表对不上，会抛出异常（底层执行会调用 method.invoke(target, args) 方法，args 对不上就挂了）。

    这个地方小册再多嘴一句，其实我们可以通过一种比较 “奇葩” 的办法，让这两个参数列表不对应，至于到底怎么搞，我们放到二次封装的环节再讲解。

    # 2. DML类Provider注解

    跟上一章一样，我们把 insert 、update 、delete 都放在一起说。

    # 2.1 @InsertProvider

    @InsertProvider 注解给了我们一种比较有趣的 insert 语句的生成方式，小册先举一个例子，各位跟着写一写。

    # 2.1.1 基本使用

        @InsertProvider(type = UserMapperProvider.class, method = "save")
    void save(User user);

    public String save(User user) {
        SQL sql = new SQL();
        sql.INSERT_INTO("tbl_user");
        sql.VALUES("id", "'" + UUID.randomUUID().toString().replaceAll("-", "") + "'");
        sql.VALUES("name", "#{name}");
        sql.VALUES("age", "#{age}");
        sql.VALUES("department_id", "#{department.id}");
        return sql.toString();
    }

    # 2.1.2 SQL构造insert的细节

    细心的小伙伴可能在自己练习的时候发现了一个细节：

    这两个方法可以传入一组可变参数，那我们是不是也可以用这种方式呢？答案是肯定的，比方说我们可以把 save 方法改为如下实现：

        public String save(User user) {
        SQL sql = new SQL();
        sql.INSERT_INTO("tbl_user");
        sql.INTO_COLUMNS("id", "name", "age", "department_id")
                .INTO_VALUES("'" + UUID.randomUUID().toString().replaceAll("-", "") + "'", 
                        "#{name}", "#{age}", "#{department.id}");
        return sql.toString();
    }

    但是这样写的话有一个弊端：可能存在一些特殊情况，它插入的数据列不固定，这样的话用 INTO_COLUMNS 和 INTO_VALUES 方法就靠不住了，毕竟这两个方法需要保证参数列表一一对应呀。所以出于灵活考虑，我们通常选择 sql.VALUES(column, value); 的方式。

    # 2.2 @UpdateProvider

    与 insert 类似，update 类的 SQL 也是非常容易构造的，而且基于编程式的 SQL 构造更具灵活性，以下是一个示例：

        @UpdateProvider(type = UserMapperProvider.class, method = "updateByExample")
    int updateByExample(User user);

    public String updateByExample(User user) {
        SQL sql = new SQL();
        sql.UPDATE("tbl_user");
        if (user.getName() != null && user.getName().trim().length() > 0) {
            sql.SET("name = #{name}");
        }
        if (user.getAge() != null) {
            sql.SET("age = #{age}");
        }
        sql.WHERE("id = #{id}");
        return sql.toString();
    }

    # 一点额外的小思考

    可能会有小伙伴产生一个疑问：为什么 update 的时候，set 没有像 insert 那样的 values 方法，传入 column 和 value 呢？ 诶，其实也可以，不过我们想一下为什么没有这么做。

    我们先回想一下 update 语句的格式：

    update tablename set column = value, column = value where id = ?
    1

    如果使用 set 语句的写法，则上面的 SQL 可以将中间的片段剪成如下样式：

    update tablename set 
[片段1], 
[片段2] 
where id = ?
    1
    2
    3
    4

    可以发现，底层只需要将一个一个的片段直接拼接起来，中间用逗号隔开即可。

    但如果是传入 column 和 value 呢？那 SQL 又应该怎么拼接呢？

    update tablename set
[column1] = [value1], 
[column2] = [value2] 
where id = ?
    1
    2
    3
    4

    有木有发现这个拼接很多余？明明这个等号你不写也是人家写，而且还多余拼两下 SQL 。与其 MyBatis 自己拼接怪费劲的，还不如直接让我们写了得了。

    那小伙伴们估计又有新的问题了：为什么 insert 的时候提供了 column 和 value 的写法呢？

    好，我们同样回忆一下 insert 语句的格式：

    insert into tablename (column1, column2) values (value1, value2);
    1

    这种格式怎么拆分和拼接呢？要么，一次性把所有的 column 和 value 都提供给 MyBatis ：

    insert into tablename ([column片段]) values ([value片段]);
    1

    要么，就提供一个 Map 集合，把每个对应的 column 和 value 都存起来，回头遍历生成 SQL 的时候再拼接：

    insert into tablename (
    for (String column : map.keys) {
        [column], 
    }
) values (
    for (String value : map.values) {
        [value], 
    }
);
    1
    2
    3
    4
    5
    6
    7
    8
    9

    这种情况下是完全可以理解的是吧，毕竟说到底，column 跟 value 没有挨在一起，那提供 column 和 value 的写法就不会不合适。

    # 2.3 @DeleteProvider

    最后我们再简单过一遍 @DeleteProvider 注解了，它的使用也没什么特殊的了，毕竟 delete 语句本身就简单嘛。

    还是按照流程来，先来定义 Mapper 接口的方法，以及 Provider 类的方法：

        @DeleteProvider(type = UserMapperProvider.class, method = "deleteById")
    int deleteById(String id);

    public String deleteById(String id) {
        SQL sql = new SQL();
        sql.DELETE_FROM("tbl_user");
        sql.WHERE("id = #{id}");
        return sql.toString();
    }
