    # 8. 责任链模式

    责任链模式，链，是不是很容易想到过滤器链？拦截器链？对的，其实过滤器、拦截器就是非常强的责任链模式的体现。

    OK ，我们还是从模式本身进行回顾。

    # 8.1 UML类图

    责任链模式的结构很清晰明了：

    但它的 UML 图却稍有复杂：

    说是复杂，其实核心还是抓住两点：1）责任链是一条链，它应该像链表那样有下一个的指针引用（ next ）；2）责任链上的每个元素都可以 / 都有机会处理业务逻辑（即 UML 上的 handle 方法）。只要这两点把握住了，责任链模式还是不难理解的。

    # 8.2 伪代码实现

    同样的，我们用伪代码来简单模拟一下上面的过滤器链：

    interface Filter {
    void handle();
    void setNext(Filter);
}

class Filter1 implements Filter {
    Filter next;
    void setNext() { ... }
    
    void handle() {
        // ......
        next.handle();
    }
}

// Filter2 同
Filter filter1 = new Filter();
Filter filter2 = new Filter();
filter1.setNext(filter2);

filter1.handle();

    # 8.3 MyBatis中的体现

    前面都提了那么多拦截器了，MyBatis 中的体现自然也就是拦截器 / 插件。在前面第 29 章中，我们看到 MyBatis 插件作用于核心组件上，是利用动态代理，依次创建代理对象：

    public Object pluginAll(Object target) {
    // 依次包装拦截器
    for (Interceptor interceptor : interceptors) {
        target = interceptor.plugin(target);
    }
    return target;
}

default Object plugin(Object target) {
    return Plugin.wrap(target, this);
}

// Plugin
public static Object wrap(Object target, Interceptor interceptor) {
    Map<Class<?>, Set<Method>> signatureMap = getSignatureMap(interceptor);
    Class<?> type = target.getClass();
    Class<?>[] interfaces = getAllInterfaces(type, signatureMap);
    if (interfaces.length > 0) {
        // 此处创建代理对象
        return Proxy.newProxyInstance(type.getClassLoader(), interfaces, new Plugin(target, interceptor, signatureMap));
    }
    return target;
}

    经过循环创建代理对象后，一条拦截器链就形成了。

    实际执行的时候，由于是一层一层动态代理创建的代理对象，它在运行时其实是跟洋葱一样一层一层调用代理的 InvocationHandler ，最终走到芯儿才是调用 target 的方法。

    多说两句，这个设计跟 SpringFramework 的 AOP 不太一样，SpringFramework 的 AOP 底层是组合了一个增强器链，那个责任链模式的体现更直观。

    # 9. 组合模式

    对于组合模式而言，可能大家接触到的最多的例子是数据结构中提到的 “树” 了吧，对于树的节点而言，它有根节点和叶子结点两种形态：根节点有叶子节点（子节点），而叶子节点没有子节点。组合模式就可以表示这种树形结构的节点，对于我们客户端而言，我们可以不用关心这个节点是单纯的一个叶子节点，还是包含很多子节点的根节点，一视同仁，对他们进行相同的操作都是没有问题的。

    另一个比较常见的场景是文件夹与文件了，我们可以直接对一个文件进行复制、剪切、粘贴等操作，也可以对一整个文件夹进行操作。

    # 9.1 UML类图

    我们先看它的 UML 类图，组合模式强调的是部分与整体的行为具有一致性，所以具体的实现都需要实现一组相同的方法：

    当然，叶子节点没有子节点了，所以上面的 UML 图中我们只标明了 update 方法，对于 add 和 remove 方法，应当在实现中予以抛出异常，代表动作不支持。

    # 9.2 伪代码实现

    照例，我们用伪代码体现一下组合模式。

    interface Component {
    void add();
    void update();
    void remove();
}

class Composite implements Component {
    List<Component> childrens;
    
    void add() { ... }
    void update() { ... }
    void remove() { ... }
}

class Leaf implements Component {
    void add() { throw new Exception ... }
    void remove() { throw new Exception ... }
    void update() { ... }
}

    # 9.3 MyBatis中的实现

    还记得第 26 章中，小册讲解动态 SQL 组合时，提到的一个 SqlNode 的概念吗？如果不记得的话，小册把那一部分的内容贴过来，小伙伴也不用去翻了。

    protected MixedSqlNode parseDynamicTags(XNode node) {
    List<SqlNode> contents = new ArrayList<>();
    // 提取出statement中所有的子节点(除了子标签之外，SQL明文也算)
    NodeList children = node.getNode().getChildNodes();
    for (int i = 0; i < children.getLength(); i++) {
        XNode child = node.newXNode(children.item(i));
        // 当前子节点是普通文本，或者xml中的CDATA类，则认定为SQL文本
        if (child.getNode().getNodeType() == Node.CDATA_SECTION_NODE || child.getNode().getNodeType() == Node.TEXT_NODE) {
            String data = child.getStringBody("");
            TextSqlNode textSqlNode = new TextSqlNode(data);
            // 普通文本也有可能是动态的
            if (textSqlNode.isDynamic()) {
                contents.add(textSqlNode);
                isDynamic = true;
            } else {
                contents.add(new StaticTextSqlNode(data));
            }
        } else if (child.getNode().getNodeType() == Node.ELEMENT_NODE) { // issue #628
            // 如果是动态SQL标签，则解析标签
            String nodeName = child.getNode().getNodeName();
            NodeHandler handler = nodeHandlerMap.get(nodeName);
            if (handler == null) {
                throw new BuilderException("Unknown element <" + nodeName + "> in SQL statement.");
            }
            handler.handleNode(child, contents);
            // 并标注当前statement是动态SQL
            isDynamic = true;
        }
    }
    return new MixedSqlNode(contents);
}

    这段逻辑是将动态 SQL 的内容封装为整体 SqlSource 的过程。它会根据一个一个的 SQL 子节点，封装为一个一个的 SqlNode （普通文本会封装为 TextSqlNode ，而动态 SQL 会让 NodeHandler 处理，也就是那些对应着动态 SQL 标签的一个一个的 Handler ，例如 IfHandler 、WhereHandler 等，这些 Handler 内部的处理也是会封装 SQL 为 SqlNode 对象）。这些封装好的 SqlNode 对象，最终会被组合进一个 List<SqlNode> 的集合 contents 中，并在方法的最后封装为一个 MixedSqlSource 。所以我们在实际 Debug 的时候，进入到 DynamicSqlSource 中，拿到的一定是一个 MixedSqlSource 类型的对象。
