    # 3. 单例模式

    单例模式，这可能是我们从一开始学习 JavaSE 的时候就了解的最经典模式了（应该是接触的第一个吧），这个模式本身怎样不用小册多说，各位也都清楚。不过按照惯例，我们还是要简单回顾一下模式本身。

    # 3.1 UML类图

    单例模式的 UML 类图估计是最简单的了吧，因为描述单例模式只需要一个类：

    我们都很熟悉了，单例模式要保证的，是通过利用一个全局变量使得一个对象可以被访问但不可被修改，同时这个对象本身的类型不对外暴露创建的方法，让这个类型本身保存唯一的对象实例，从而达到单实例的目的。

    # 3.2 伪代码实现

    有关单例模式的代码实现，我们已经写得太多了，比较常见的方式是懒汉和饿汉形式：

    // 懒汉单例模式，侧重于省初始化的空间资源(等到用的时候才创建)
public class Singleton {
    private static Singleton instance = null;
    
    private Singleton() {
        System.out.println("Singleton constructor run ......");
    }
    
    public static Singleton getInstance() {
        if (instance != null) {
            return instance;
        }
        synchronized (Singleton.class) {
            if (instance == null) {
                this.instance = new Singleton();
            }
            return instance;
        }
    }
}
// 饿汉单例模式，侧重于省运行时计算资源(不用加锁判断)
public class Singleton {
    private static Singleton instance = new Singleton();
    
    private Singleton() {
        System.out.println("Singleton constructor run ......");
    }
    
    public static Singleton getInstance() {
        return instance;
    }
}

    # 3.3 MyBatis中的体现

    MyBatis 中涉及到的单例模式不算很多，但是有一个设计还蛮经典的，我们可以了解一下，就是分布在源码中的这个 ErrorContext 错误上下文：（只保留关键成员）

    public class ErrorContext {

    // 饿汉单例模式
    private static final ThreadLocal<ErrorContext> LOCAL = ThreadLocal.withInitial(ErrorContext::new);

    // ......
    
    private String sql;
    private Throwable cause;

    private ErrorContext() {
    }

    public static ErrorContext instance() {
        return LOCAL.get();
    }

    可以发现，这个设计其实很 “单例模式” 了，只不过这里多了一层设计：ErrorContext 为每一个线程准备了一个实例，所以其实这不是严格意义上的单实例。不过对于特定的某个线程来讲，它只能拿到属于自己线程的 ErrorContext ，用于记录整个执行期间发生的异常信息，也是完全没有问题的、

    可能有的小伙伴没见过 ThreadLocal.withInitial(ErrorContext::new) 这种写法，有点懵，其实它就是在创建 ThreadLocal 时，先默认传入一个对象，这样 ThreadLocal 创建完成的时候，内部也就有值了，这即是饿汉单例模式的体现。

    # 4. 代理模式

    代理模式也是我们非常非常熟悉的了，代理模式包含静态代理和动态代理，我们在框架中看到的大多都是动态代理了，不过我们还是分别对这两种代理回顾一下。

    # 4.1 UML类图

    对于代理模式来讲，它的设计是提供一个与目标对象具有同样行为的代理，让代理来控制目标对象的引用，进而控制目标对象的行为。代理模式的 UML 图本身不难，仅有一个定义行为的接口，以及目标类和代理类：

    动态代理的设计与静态代理一致，我们都知道，原生的 jdk 动态代理是需要被代理的对象所属类必须有至少一个接口的实现，才能使用动态代理创建对象，所以最终的最终，还是需要有接口才 OK 的。

    Cglib 动态代理不在我们的考虑范围之内，因为其底层运用字节码增强的方法，本身就不是使用代理模式的设计模型，所以我们不对此展开讨论。

    # 4.2 伪代码实现

    代理模式，在代码上的实现本身也不难，按照上面的 UML 类图即可轻松编写。我们可以假设一个场景：明星接广告，和经纪人接广告。

    interface Subject {
    void receiveAds();
}

class Celebrity implements Subject {
    void receiveAds() {
        print("明星接广告了~");
    }
}

class Broker implements Subject {
    
    private Celebrity target;
    
    public Broker(Celebrity target) {
        this.target = target;
    }
    
    void receiveAds() {
        print("经纪人代替明星接广告了~");
    }
}

    # 4.3 MyBatis中的体现

    MyBatis 中的代理模式那当属 Mapper 动态代理了，这个特性那是相当的重要了！

    # 4.3.1 Mapper动态代理

    Mapper 动态代理的套路，其实就是利用了 jdk 动态代理中的 InvocationHandler ，它可以拦截空接口的动作，执行增强的逻辑。MyBatis 底层给 Mapper 接口创建代理对象的方式也算是比较朴素了，就是拿到接口后，直接塞一个 InvocationHandler 的实现类 MapperProxy ，利用 Proxy.newProxyInstance 创建出代理对象，返回，搞定。

    public <T> T getMapper(Class<T> type, SqlSession sqlSession) {
    final MapperProxyFactory<T> mapperProxyFactory = (MapperProxyFactory<T>) knownMappers.get(type);
    if (mapperProxyFactory == null) {
        throw new BindingException("Type " + type + " is not known to the MapperRegistry.");
    }
    try {
        return mapperProxyFactory.newInstance(sqlSession);
    } // catch ......
}

public T newInstance(SqlSession sqlSession) {
    final MapperProxy<T> mapperProxy = new MapperProxy<>(sqlSession, mapperInterface, methodCache);
    return newInstance(mapperProxy);
}

protected T newInstance(MapperProxy<T> mapperProxy) {
    return (T) Proxy.newProxyInstance(mapperInterface.getClassLoader(), new Class[] { mapperInterface }, mapperProxy);
}

    # 4.3.2 插件/拦截器

    再一个代理模式的体现，是插件 / 拦截器的使用了。我们在生命周期的自定义插件章节看过，底层都是使用 jdk 动态代理，为那 4 个可以增强的对象创建代理对象：

    // Plugin
public static Object wrap(Object target, Interceptor interceptor) {
    Map<Class<?>, Set<Method>> signatureMap = getSignatureMap(interceptor);
    Class<?> type = target.getClass();
    Class<?>[] interfaces = getAllInterfaces(type, signatureMap);
    if (interfaces.length > 0) {
        // 使用动态代理增强4大对象
        return Proxy.newProxyInstance(type.getClassLoader(), interfaces, new Plugin(target, interceptor, signatureMap));
    }
    return target;
}
