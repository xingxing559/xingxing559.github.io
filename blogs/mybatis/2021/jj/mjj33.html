    # 5. 模板方法模式

    模板方法模式算是我们见过的比较多而且容易理解的设计模式了，我们平时做通用抽取和逻辑抽象时也会用到。下面我们还是先来回顾一下。

    # 5.1 UML类图

    模板方法模式的经典实现是一个抽象的父类 + n 个子类，父类用于编写逻辑骨架，子类在特定的模板方法上负责实现。这里我们可以举一个例子：小龙虾的多种制作方法。

    先看一下 UML 图：

    在这个设计中，父类的小龙虾只是一个抽象的制作过程，里面定义了清洗、准备辅料、初加工、烧制等方法，其中准备辅料和烧制，需要根据不同做法区别准备，所以这两个方法是抽象方法，也即模板方法。下面的两个子类是具体的两种做法，麻辣小龙虾要准备的是与辣椒相关的辅料，而蒜蓉小龙虾则要准备以蒜蓉为主的调味料，烧制的方法也是不一样的。这就体现出来模板方法模式的核心特性了：父类定义方法的逻辑骨架，而其中一些逻辑步骤的实现放到具体的子类中实现，这样可以做到不改变逻辑结构的前提下冲定义某些细节的实现。

    # 5.2 伪代码实现

    按照上面的 UML 图，我们可以写出对应的伪代码：

    abstract class 小龙虾 {
    void 制作() {
        清洗();
        准备辅料();
        初加工();
        烧制();
    }
    
    void 清洗() {
        print("清洗小龙虾。。。");
    }
    
    abstract void 准备辅料();
    
    void 初加工() {
        print("过油炒制。。。");
    }
    
    abstract void 烧制();
}
class 麻辣小龙虾 extends 小龙虾 {
    void 准备辅料() {
        print("准备姜片、香料、辣椒、花椒、红油。。。");
    }
    
    void 烧制() {
        print("使用清水、高度白酒烧制小龙虾。。。");
    }
}
class 蒜蓉小龙虾 extends 小龙虾 {
    void 准备辅料() {
        print("准备蒜蓉、姜片、红彩椒。。。");
    }
    
    void 烧制() {
        print("使用啤酒烧制小龙虾。。。");
    }
}

    # 6. 适配器模式

    适配器模式，可能会有些小伙伴认为跟代理有点像，但适配器跟代理还是不一样的。

    适配器要做的事，是让一个跟你对不上口的件儿，通过一个中间件保证能让你用上；而代理要做的，是在原有的件儿上增强，或者控制原有件儿的行为。说得再简单点，适配器是让你用的件儿跟原来一样，代理是不指定让你用到更好的还是更差的件儿。

    OK ，说回整体，我们还是重点回顾适配器本身的设计。

    # 6.1 UML类图

    生活中我们见到的适配器就非常多了，一个经典的设计便是电源适配器。我们用的笔记本电脑，轻薄本大多都是 65W 或者 95W 的，它们接受的电压都是十几伏，但家用的照明电压都是 220V ，这可咋整呢？于是就需要使用一个电源适配器来转换一下电压，让笔记本能正常的接电。

    我们可以把这个场景用 UML 图表示出来，适配器的 UML 类图本身不算很难：

    它强调的重点，是让一个类的接口，转换为另一个可以让客户端使用的接口，换句话说，它可以使得让原来那些不兼容的组件一起参与工作。上图中，Laptop 本身不能在 220V 的电压下运行，但我们使用一个 PowerAdapter ，让它去组合 Laptop ，并将 220V 的电压转换为 65W 的功率，这样 Laptop 就可以正常工作了。

    # 6.2 伪代码实现

    既然上面我们都把 UML 图画出来了，那下面我们就可以用伪代码实现一下了：

    interface OrdinaryVoltageDevice {
    void useAt220V();
}
class Television implements OrdinaryVoltageDevice {
    void useAt220V() {
        print("电视在220V的电压下运行。。。");
    }
}
class Laptop {
    void useAt65W() {
        print("笔记本在65W的功率下运行。。。");
    }
}
class PowerAdapter implements OrdinaryVoltageDevice {
    Laptop laptop;
    constructor() { ... }
    
    void useAt220V() {
        print("电压转换完毕。。。");
        laptop.useAt65W();
    }
}

    # 7. 装饰者模式

    装饰者模式，又是一个跟适配器模式有点像的模式，不过装饰者要做的事情更简单：它只负责给原有的对象添加新的功能，简而言之：装饰者只会让原本的对象行为变得更复杂。

    # 7.1 UML类图

    装饰者的核心，是所有要装饰的特性都实现同一个装饰接口，这个接口本身又有一个基础的落地实现。所以它的 UML 图相对会复杂一些。

    这里我们以人穿衣服为例，人和衣服本身都属于可被装饰的对象，人先穿衣服，然后衣服继续套衣服，从而形成装饰的效果。这也就体现了装饰者模式最大的好处：可以动态任意组合不同的装饰效果，相比较于继承来讲更加灵活。

    # 7.2 伪代码实现

    同样的，我们用一段伪代码来体现一下上面的装饰者模型。

    interface CanBeDecorated {
    void show();
}
class Person implements CanBeDecorated {
    void show() {
        print("Hello, 看我！");
    }
}
abstract class AbstractClothes implements CanBeDecorated {
    CanBeDecorated target;
    constructor() { ... }
    
    void show() {
        doShow();
        target.show();
    }
    
    abstract void doShow();
}
class Tshirt extends AbstractClothes {
    void doShow() {
        print("T恤~");
    }
}
class Suits extends AbstractClothes {
    void doShow() {
        print("西装~");
    }
}

    # 7.3 MyBatis中的体现

    MyBatis 中体现的装饰者模式，最最经典的当属缓存模型。还记得吗，那个 Cache 接口的一大堆实现类：

    大多数的类都在一个叫 decorator 的包下，属实是装饰者的体现了。

    在具体的缓存构建时，最终的代码体现可以在

    private Cache setStandardDecorators(Cache cache) {
    try {
        MetaObject metaCache = SystemMetaObject.forObject(cache);
        if (size != null && metaCache.hasSetter("size")) {
            metaCache.setValue("size", size);
        }
        if (clearInterval != null) {
            // 装饰者
            cache = new ScheduledCache(cache);
            ((ScheduledCache) cache).setClearInterval(clearInterval);
        }
        if (readWrite) {
            // 装饰者
            cache = new SerializedCache(cache);
        }
        // 装饰者
        cache = new LoggingCache(cache);
        // 装饰者
        cache = new SynchronizedCache(cache);
        if (blocking) {
            // 装饰者
            cache = new BlockingCache(cache);
        }
        return cache;
    } // catch ......
}

    这样设计的意义我们也回顾一下吧，MyBatis 中的二级缓存本身是占用应用空间的，换句话说，MyBatis 中的二级缓存实际使用的是 JVM 的内存，那默认情况来讲，占用的内存太多不利于应用本身的正常运行，所以 MyBatis 会针对缓存的各种特性和过期策略，设计了一些能够修饰原本缓存件的装饰者，以此达到动态拼装缓存实现的目的。
